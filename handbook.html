Performance Workspace — Handbook

How the app works, how to run it, and how to hand it over. Version 1 — July 2026.

1. What this is

One web application that merges the two department prototypes — the Tech Stack and the Research Hub — into a single shell with four pages: Overview, Knowledge Base, Technology, and Research Hub. Everyone signs in through the same front door; what they can see and do depends on their role.

The app is a static site: a folder of HTML and JavaScript files. There is no server of our own. All live data (equipment, bookings, ideas, events, posts, quiz scores) sits in Google Firebase — specifically Cloud Firestore, the same database project both prototypes already used (phd-tracker-22d93). Nothing was migrated; the merged app reads and writes the very same collections, so existing data appeared in the new app on day one.

2. Why this architecture

A static site + Firebase is the simplest stack that still gives you a real product: no servers to patch, no backend code to maintain, free hosting tier, and real-time updates (change something in the admin, everyone's screen updates live). It is also the easiest to hand over — any web developer can read it, and Firebase is one of the most widely known platforms in the industry.

Today, the sign-in screen has a "Preview as" role picker — a deliberate mock so you can check what each audience sees while it's cheap to change. Next phase, Firebase Authentication (email signup) replaces it: new accounts land as "pending"; a super admin approves them and assigns a role. Later, PSG SSO is a switch inside the same Firebase Auth — no rebuild.

4. The pages

Overview — landing page: quick links into the three areas plus live cards (next research events, latest Tip of the week, upcoming equipment bookings) fed straight from the database.

Knowledge Base — structure in place (process areas, SOP cards, statuses); content to be filled in. Wiring it to Firestore so admins can publish articles is a small follow-up.

Technology — the tech stack catalog (search, domain / owner / data-type filters, grid & table views), each item opens with full details, a booking form, and that item's upcoming bookings. A Bookings tab shows the month calendar and all upcoming bookings. Area admins get a Manage tab (add / edit / delete equipment).

Research Hub — all eight sections of the original hub: overview, submit idea, calendar, papers (live PubMed feed with star ratings and PDF requests), PDF library, Tip of the week, PhD activities (PIN-gated per student), and the weekly Quiz. Research admins additionally get the Admin desk: idea screening, PDF request fulfilment, and event / tip / library / quiz management.

The papers feed itself is not stored: it is fetched live from PubMed RSS (via the rss2json service), exactly as the prototype did.

6. The road to production

Step 1 — Review this build. Click through every role with the "Preview as" picker. Anything wrong with who-sees-what is a one-line change now.

Step 2 — Real accounts (Firebase Authentication). Enable Email/Password in the Firebase console; the sign-in screen swaps the role picker for signup/login; a users collection stores each account's role and approval state; super admins get a small Users screen to approve and assign roles. Why: identity has to come from a system, not a picker, before anything is enforceable.

Step 3 — Security rules. Firestore rules are the actual lock: today the database accepts writes from anyone who finds it (that is how the prototypes worked). Rules will say, e.g. "only signed-in, approved users may read; only the Technology area admin may write to techStack." Hiding a button is cosmetics; rules are enforcement. This step is non-negotiable before go-live.

Step 4 — Hosting. Firebase Hosting: install the Firebase CLI once (npm install -g firebase-tools), then firebase login, firebase init hosting (pick the existing project), copy the app files into the public folder, and firebase deploy. You get an https URL immediately; a custom domain is a DNS record away.

Step 5 — SSO later. Enable the Microsoft (or Google) provider in Firebase Auth and add one button. Accounts, roles, and rules all stay as they are.

7. Handover notes (for a developer, if ever needed)

Plain React (no build step — Babel transpiles in the browser via hub-loader.js, which lists every source file in load order). Key files:

  • AppShell.dc.html — entry point; hub.css/ds-base.js — design tokens
  • hub-firebase.js — the single database layer (all reads/writes go through window.HubDB)
  • hub-i18n.jsx — EN/FR/IT/ES strings; hub-main.jsx — app root, routing, roles
  • hub-screens-*.jsx — one file per area (home, technology, research, research-extra)

Known simplifications, on purpose for phase 1: the PhD PINs and role gates live in the client (enforce with Auth + rules in phases 2–3); tip-post cover images are pasted URLs rather than file uploads; the quiz week rotates over a 30-question cycle keyed to the ISO week.
